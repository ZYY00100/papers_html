01
INTRODUCTION · 0:00

What prompt engineering actually is

提示词工程到底是什么

视频第 1 章只有 1 分钟多，但 Hannah 用一句话把整个 workshop 的核心定义给出来了：

HANNAH · 原句 Prompt engineering is the practice of writing clear instructions for the model, giving the model the context it needs to complete the task, and thinking through how we want to arrange that information in order to get the best result.

三个动作，缺一不可：

  • 清晰的指令 —— 你要 Claude 做什么？动词明确、对象明确、约束明确。
  • 必要的上下文 —— Claude 没有你的背景知识。你脑子里的"显然"，它都不知道。
  • 合理的信息排列 —— 信息怎么放是有讲究的。先后顺序、静态 vs 动态分组。

这个定义里有几个关键词值得拎出来：

  • clear instructions —— "清晰"不是"详细"。详细但不清晰（如堆砌形容词）反而更糟。
  • the context it needs —— 注意是"它需要的"，不是"你知道的"。该省的省。
  • arrange that information —— "排列"是个动词。这意味着顺序很重要。

Christian 紧跟着补了一句："The best way to learn this is just to practice doing it."（学这个最好的办法就是练）。提示词工程不是看书能学明白的——你得写、改、看输出、再改。

普通读者怎么用
把这个定义抄下来贴在你 Claude Projects 的 system prompt 顶部。下次写 prompt 时过一遍：(1) 我说清楚了吗？(2) 该给的上下文给了吗？(3) 信息排列合理吗？三个都回答了再发。
02
SETTING THE STAGE · 1:28

Why we use a real scenario

真实场景的价值 —— V1 的"s skiing accident"幻觉

第 2 章视频里只有 2 分钟，但这是 workshop 最关键的"心理锚点"：用真实场景，而不是 toy example。

场景设定是这样的：一家瑞典保险公司收到一份瑞典语的车祸报告单（带 Vehicle A 在左栏、Vehicle B 在右栏）+ 一张手绘草图。需要 Claude 判断责任归属。

为什么选这个场景？三个理由：

  • 真实业务场景 —— 不是"让 Claude 写首诗"这种 toy example。是生产环境会遇到的真实任务。
  • 多模态输入 —— 既有结构化表单（checkbox），又有非结构化图像（手绘草图）。考验 Claude 的视觉理解 + 语言理解。
  • 容易出错 —— 瑞典语 + 车牌 + 街道名，模型很容易"幻觉"出熟悉的名字。出错点足够多，能演示怎么修。

讲者当场演示了 V1 朴素 prompt 跑出来的结果——Claude 把车祸报告"读成了滑雪事故"。

V1 · 朴素 prompt 的结果 Claude hallucinates a "skiing accident" because the Swedish street name rings a bell. The model is confidently wrong — and that's the danger.

这个"s skiing accident"是整个 workshop 的核心隐喻。模型会自信地胡说。它不会说"我不确定"——它会编一个听起来合理的答案。你的 prompt 设计工作不是"问对问题"，是"让模型不可能编"。

普通读者怎么用
想想你自己最近一次被 Claude 误导的场景。是它"自信地错了"，还是"不知道答错了"？如果是前者——你下次需要在 prompt 里加"如果不确定，明确说我不确定"。这是 task context 里"设定回答边界"的具体应用（见第 4 章）。
03
PROMPT STRUCTURE · 3:57

The 10-step prompt anatomy

10 步 prompt 解剖图 —— 一个完整的 prompt 长什么样

第 3 章是整个 workshop 的骨架。Hannah 把一个完整的 prompt 拆成 10 个部分，按顺序排列：

<role>You are a claims adjuster at a Swedish insurance company.</role>
<tone>Be factual and confident. Do not guess.</tone>
<background>Below is the accident report form and the sketch.</background>
<task>Determine which driver is at fault based on the form and sketch.</task>
<rules>If you cannot determine fault confidently, say "Insufficient information".</rules>
<examples>
  <example>
    Input: [form + sketch where Vehicle A ran a red light]
    Output: Vehicle A is at fault. Reason: ran red light.
  </example>
</examples>
<conversation_history>[if any]</conversation_history>
<request>Analyze this case: [form + sketch]</request>
<thinking>Think step by step before answering.</thinking>
<output_format>Respond in XML: <verdict>...</verdict> <reasoning>...</reasoning></output_format>
<prefill>Sure, here's my analysis: <verdict>  # assistant role pre-fill

这 10 个部分不是死规定——根据任务可以增减。但顺序是固定的：

  1. 角色（role / task context） —— 谁是 Claude / 它扮演什么
  2. 语气（tone context） —— 声音风格
  3. 背景信息（background data, documents & images） —— 原始材料
  4. 任务与规则（detailed task description & rules） —— 做什么 / 不做什么
  5. 示例（examples） —— 用 XML tag 包装的 few-shot
  6. 对话历史（conversation history） —— 相关历史
  7. 具体请求（immediate task / request） —— 当前问题
  8. 思考（think step by step） —— 先想再答
  9. 输出格式（output formatting） —— XML / JSON / Markdown
  10. 预填充（prefilled response） —— 助手起始字符串

关键原则：静态信息放 system prompt，动态信息放 user 消息。"你是一个保险理赔员"是每次会话都不变的，放 system prompt；"今天这份报告长这样"是每次都变的，放 user 消息。这样做的好处：

  • 节省成本 —— system prompt 的内容会被 prompt caching 命中，每次只算一次。
  • 推理更稳 —— 模型推理时优先处理 system prompt（长期上下文），再处理 user 消息（当前任务）。
  • 维护简单 —— 改 system prompt 一次，所有会话生效。
HANNAH · 原句 The order of your prompt matters more than the words in it. Static context (role, background, rules, examples) before the dynamic request. This helps Claude reason better and cache more of the prompt.
普通读者怎么用
把上面 10 步结构当成 prompt 模板。你写任何 prompt 时，过一遍这 10 步。哪些需要哪些不需要就留下对应的。这样你写 prompt 的速度会变快，质量会变稳。
04
TASK CONTEXT · 5:40

Who Claude is and when to stay silent

任务上下文 —— 让 Claude 知道自己是谁、什么时候该闭嘴

第 4 章讲 prompt 的"开篇三件套"：角色（role）、语气（tone）、回答边界（when not to answer）。这三条放在一起，决定了 Claude 在整个对话里的"工作姿态"。

讲者在视频里现场把 V2 prompt 改了——加上了："This AI system assists a human claims adjuster reviewing Swedish car accident report forms."Claude 立即停止幻觉成"s skiing accident"。

V1 → V2 · 关键变化 加入 task context 后，Claude 知道自己"是保险理赔员，看的是车祸报告"——瑞典街道名突然不再触发滑雪联想。Task context 是一种先验约束，让模型从一开始就在正确的语义空间里推理。

具体的几个技巧：

  • 明确角色定位 —— 不要只说"helpful assistant"。说"你是一名资深瑞典保险理赔员，专门处理车祸定责"。具体到角色、行业、专业领域。
  • 指定语气 —— "factual and confident" / "warm and conversational" / "concise and direct"——明确告诉 Claude 声音长什么样。
  • 设定回答边界 —— "If you are not fully confident in your assessment, do not make one." 这一条让 Claude 从"自信胡说"变成"不确定就拒答"。
  • 不要让 Claude 演你不想要的角色 —— 没指定角色，Claude 会"猜你想要什么"。猜错了就翻车。
CHRISTIAN · 原句 Tell Claude what role it's playing, what tone it should use, and when to stay silent. Three lines, three different failure modes fixed.
普通读者怎么用
打开你最近用的 prompt。第一句话是不是 "You are a helpful assistant"？如果是，改掉。换成 "You are a [具体角色] helping with [具体任务]"。加一句语气 + 一句"什么时候不要答"。你会立刻看到输出变稳。
05
BACKGROUND DETAIL · 8:47

Static vs dynamic, and where to put long context

背景细节 —— 静态 vs 动态、长文档怎么放

第 5 章是关于怎么给 Claude 喂材料。看似简单，其实有几条不显然的规则。

第一条规则——区分静态 vs 动态信息：

  • 静态信息 —— 每次会话都不变的内容（角色、规则、风格指南、领域知识）。放 system prompt。
  • 动态信息 —— 每次都变的内容（当前问题、当前文档、当前任务）。放 user 消息。

为什么这样分？省钱 + 推理更稳。Anthropic 的 prompt caching 会把 system prompt 的开头部分缓存起来——static 内容越多，cache hit rate 越高，成本越低。

第二条规则——文档数据放 user 消息。但有讲究：

ANTHROPIC DOCS · 最佳实践 Put longform data at the top: Place your long documents and inputs near the top of your prompt, above your query, instructions, and examples. This can significantly improve performance across all models.

翻译成实际动作：

  • 长文档放顶部 —— 20k+ tokens 时尤其重要。长文档在 prompt 顶部，query 在底部。
  • 多文档按相关性排 —— 最相关的放最前面，让 Claude 优先关注。
  • 图像可以 Base64 编码 —— 直接放进 user 消息的 content 数组里。Claude 能理解图像。
  • 多图像场景下 Claude 表现很好 —— Opus 4.5/4.6 在多图像理解上有专门的优化。视频帧也可以拆成图传。
  • 用 XML tag 划分背景信息 —— <document_1> <document_2> 让 Claude 知道哪是哪。
<document_1>
[Vehicle A 报告内容]
</document_1>

<document_2>
[Vehicle B 报告内容]
</document_2>

<sketch>
[手绘草图，Base64 编码]
</sketch>
普通读者怎么用
把 Claude Projects 的 system prompt 当成"公司手册"——所有不变的内容都放进去：你的品牌声音、用户画像、领域术语表、必遵守的规则。每次新对话就只需要写 user 部分。长期下来省很多钱。
06
EXAMPLES · 12:55

Few-shot beats many rules

少样本学习 —— 几个例子胜过十页规则

第 6 章讲 few-shot prompting——给 Claude 看几个示例。Anthropic 在这件事上的判断很鲜明：

ANTHROPIC · EFFECTIVE CONTEXT ENGINEERING For an LLM, examples are the "pictures" worth a thousand words.

意思很清楚：示例比规则有效得多。给 Claude 看 3 个"该怎么做"的例子，比写 10 条"必须遵守的规则"有用得多。

讲者在视频里演示了 V3 → V4 的飞跃——加了一组带 annotation 的示例（标注每个 checkbox 对应的责任判定），Claude 的输出从"60% 准确"提升到"几乎 100% 准确"。

具体的几个技巧：

  • 少量多样化示例 —— 不要堆 50 个相似示例。3-5 个覆盖典型场景的示例胜过 50 个重复的。
  • XML tag 包装示例 —— 用 <example> 包裹输入和输出，让 Claude 清楚这是"学习样本"，不是"当前任务"。
  • 边缘案例做成带标注的示例 —— 那些你"不确定 Claude 会不会懂"的边界场景，做成示例教它。不要堆进规则清单。
  • 不要堆砌所有 edge case —— 错误做法是"把所有可能出错的场景都列出来"。正确做法是挑最有代表性的几个做成示例。
  • 示例放在请求前 —— 顺序很重要：示例 → 当前请求，不是反过来。Claude 先看"该怎么做"再看"这次怎么做"。
<examples>
  <example>
    <input>Vehicle A ran red light, Vehicle B had right of way</input>
    <output>Vehicle A at fault</output>
  </example>
  <example>
    <input>Both vehicles had green light; unclear who entered first</input>
    <output>Insufficient information</output>
  </example>
</examples>

<request>Now analyze: [current case]</request>
普通读者怎么用
写 prompt 时问自己："我能不能用 3 个例子替代 10 条规则？"大多数情况答案是 yes。把规则删掉，换成示例。Claude 的输出会更稳，你也更容易维护（改示例比改规则直观）。
07
CONVERSATION HISTORY · 15:02

What to carry over, what to drop

对话历史 —— 多轮里传什么、不传什么

第 7 章很短，但解决一个多轮对话的核心问题：历史消息怎么传。

Claude 在多轮对话里默认能看到所有历史消息。但"能看到"不等于"该让看到"。

几个关键原则：

  • 传相关历史，不传全部 —— 多轮应用里，把"和当前任务相关的"前几轮传进去。如果用户切换话题，开新对话。
  • 摘要历史比贴原历史省 token —— 超过 10 轮后，把早期消息摘要成 2-3 段。最近几轮保留原话。
  • 历史相关性过滤 —— 工具调用返回的中间结果可以摘要；用户表达的核心意图保留原话。
CHRISTIAN · 原句 Don't pass everything. Pass what's relevant. Summary > raw when the conversation is long.

讲者在视频里演示了一个反例：把整个对话历史传给 Claude，让它"基于之前讨论的修改方案"，结果 Claude 被早期废弃的方案干扰，输出了错误结论。历史太长不是优势，是噪声。

普通读者怎么用
在 Claude Projects 里写一个 "上下文摘要" workflow：对话超过 10 轮时，先用一条 prompt 总结"目前确定下来的事"和"待解决的事"，然后把这份摘要带进下一轮对话。这比贴原历史清晰得多。
08
TASK REMINDER & STEP-BY-STEP · 15:55

Recap at the end, think before answering

任务提醒 + 分步指令 —— 末尾 recap，先想再答

第 8 章是 workshop 最反直觉的一节。两个看似矛盾的动作：

第一个动作 —— 任务末尾再 reminder 一次。

Claude 的注意力在长 prompt 里会"前重后轻"。把核心规则在 prompt 末尾再讲一遍，让 Claude 在生成回答前最后看一眼。

CHRISTIAN · 原句 Recap the task at the end. Don't assume Claude will remember the rules from 500 lines ago.

具体做法——在 prompt 的最后部分加一段 <important> 或 <reminder>，把最重要的规则复述一遍：

[前面的所有内容...]

<reminder>
- 只输出 <verdict> 和 <reasoning> 两个 XML 标签
- 不确定时回复 "Insufficient information"，不要瞎猜
- 用英文回答
</reminder>

第二个动作 —— Think step by step。

让 Claude 在给最终答案前先"出声思考"。把思考过程和最终答案用 XML tag 分开：

<thinking>
Let me analyze each checkbox:
- Vehicle A checkbox "ran red light": YES
- Vehicle B checkbox "had right of way": YES
- So Vehicle A is at fault.
</thinking>

<verdict>Vehicle A</verdict>
<reasoning>Vehicle A ran red light while Vehicle B had right of way.</reasoning>

这条技巧的机制很清晰：

  • 末尾 reminder 抗衰减 —— 长 prompt 后面 Claude 的注意力会下降，reminder 把核心规则重新激活。
  • Think step by step 提升复杂任务准确率 —— 推理前显式思考比直接答更准。CoT（Chain of Thought）的核心。
  • 用 XML tag 区分思考和答案 —— <thinking> 给过程，<answer> 给结果。程序好解析，调试好看。
  • 步骤顺序符合人类推理 —— 不要写"先结论再分析"。写"先分析 checkbox、再对照规则、最后给结论"——按人脑思考顺序。
普通读者怎么用
给任何长 prompt（超过 200 字）加一个 <reminder> 块——3-5 条最重要的规则复述。同时加 <thinking> 要求让 Claude 先思考再答。这两条加起来准确率提升 10-30%。
09
OUTPUT FORMATTING · 20:05

Structure your output so apps can parse it

输出格式 —— 让程序能直接解析

第 9 章讲输出格式控制。这条对工程化使用 Claude 至关重要——你不会想每次手动从 Claude 的回复里"抠出"你想要的数据。

讲者在视频里演示了 V5——给 Claude 套上 XML 输出约束：

<output_format>
Always respond using this exact structure:
<verdict>Vehicle A or Vehicle B or "Insufficient information"</verdict>
<reasoning>Brief explanation based on the form and sketch.</reasoning>
</output_format>

关键技巧：

  • 明确指定输出格式 —— XML / JSON / Markdown / 表格。越具体越好。
  • 让程序能直接解析 —— 输出 XML 标签包裹的数据，下游代码可以直接 parse。这是工程化 Claude 的基础。
  • XML 嵌套表达层级 —— <user><name>...</name><age>...</age></user> 比平铺 2 字段更清晰。
  • 格式说明放后段 —— 角色 / 任务 / 上下文 / 思考在前，输出格式在后。模型先理解任务，再被约束输出格式。
CHRISTIAN · 原句 Tell Claude exactly what shape you want the answer in. XML and JSON are best because they parse cleanly. If you let Claude free-form, you'll spend more time parsing its answer than getting value.

实际效果：V5 之前 Claude 的输出是"Vehicle A appears to be at fault based on..."——一段话，要程序去解析。V5 之后是干净的 <verdict>Vehicle A</verdict>，程序一行代码就能取出来。

普通读者怎么用
如果你把 Claude 的输出接到任何下游工具（Notion API、Google Sheets、自动化脚本），一定要在 prompt 里明确输出格式。用 XML 标签或 JSON 结构。这是 Claude "工程化"和"玩具用"的分水岭。
10
PRE-FILL & EXTENDED THINKING · 22:15

Start the answer yourself, and let Claude think out loud

预填充 + 扩展思考 —— 替 Claude 起个头，让它"出声想"

第 10 章是 workshop 的最后两个技巧，这两个技巧在 Claude API 里能用，普通网页 Claude 聊天框里部分能用。

第一个 —— 预填充（Pre-fill）。

在 API 调用时，你可以在 Assistant 角色里先写一段字符串作为"开头"。Claude 接着你的开头继续写。

// API 调用示意
messages: [
  { role: "user", content: "分析这份车祸报告" },
  { role: "assistant", content: "<verdict>" }  // 预填充
]

为什么这样做？

  • 强制输出格式 —— 预填充 <verdict>，Claude 不会先写"以下是分析结果："这种废话。
  • 跳过模型自动写的前缀 —— 模型默认会写"Sure!" / "Of course!" / "Let me help you with that."——预填充直接进入正题。
  • 用预填充控制语气起头 —— 预填充 "Here's my analysis:"——Claude 接下来就是分析的语调。

第二个 —— Extended Thinking。

Claude 的扩展思考模式让它"出声思考"——把推理过程显式写出来再给答案。这和前面说的 "Think step by step" 不同：

  • "Think step by step" 是 prompt 技巧 —— 你写一句话让 Claude 思考。它遵守或不遵守取决于 prompt。
  • Extended Thinking 是模型能力 —— 你在 API 里启用，Claude 内部先做大量推理再生成答案。质量更高、更慢。
  • 可视化推理 —— Extended Thinking 的输出包含完整的思考过程，调试时直接看推理链路。
  • 适合调试 + 复杂任务 —— 模型答错时，看 Extended Thinking 知道它哪里想错了；任务复杂时（多步骤推理），Extended Thinking 准确率高很多。
HANNAH · 原句 Pre-fill skips the small talk. Extended thinking shows you the work. Both make debugging Claude dramatically easier.
普通读者怎么用
Pre-fill：在 Claude.ai 网页聊天框里，把 Claude 上一次的回复修改一下再发送——这等于手动预填充。Extended Thinking：在 Claude API 里启用 thinking={"type": "enabled"} 参数。两者结合：Extended Thinking 推理完，预填充 <final_answer> 让 Claude 直接给结论。

BONUS · 配套视频

Tooling and agents (from Prompting for Agents)

配套福利 —— 工具与 Agent 的提示词要点

Prompting 101 主讲完后，Anthropic 还发布了一支姊妹篇：Prompting for Agents（29 分钟，讲 agent 怎么写 prompt）。那支视频里有两条和工具相关的关键技巧，纳入这 40 个里：

技巧 39 — 工具描述要清晰独立。

给 agent 接一堆工具时，每个工具的描述要让模型知道什么时候该用它。讲者在视频里举例：

CHRISTIAN · 原句 Don't give Claude six search tools that all search slightly different databases. That confuses the model. Keep tools distinct and combine similar ones into one.

反例——给 agent 接 6 个搜索工具，每个搜索不同的数据库。这会让 Claude 困惑"该用哪个"。

正例——把 6 个搜索合并成 1 个，根据参数决定搜哪个。或者保持 6 个但每个的描述非常明确地说明触发条件。

技巧 40 — 工具错误信息要 actionable。

工具返回错误时，不要只返回 stack trace 或 error code。返回告诉 agent 下一步该怎么改的错误信息。

ANTHROPIC · WRITING TOOLS FOR AGENTS If a tool call raises an error, prompt-engineer your error responses to clearly communicate specific and actionable improvements, rather than opaque error codes or tracebacks.

反例："Error: 400 Bad Request"。

正例："The 'date' parameter must be in ISO 8601 format (YYYY-MM-DD). You provided '2025/01/15'. Please retry with '2025-01-15'."

普通读者怎么用
如果你在用 Claude Code / Cursor / 任何接 agent 的工具——看看那些工具的描述是不是"清楚独立、错误信息 actionable"。好工具 = Claude 用得对；烂工具 = Claude 反复撞墙。这不是 Claude 的问题，是工具设计的问题。

THE META-LESSON

5 principles that run through all 40 techniques

贯穿全程的 5 条元原则 —— 40 个技巧背后的 5 件事

前面 11 章拆了 35 个具体技巧。但技巧清单本身不是最重要的——技巧背后的元原则才是。以下 5 条贯穿全程：

  • 提示工程是迭代的实证过程 —— 不是"写一次就完事"。是写 → 跑 → 观察 → 修 → 再跑。V1 → V5 的过程就是这条原则的演示。
  • 不要信任模型，要为它设计 —— Claude 默认会出错。你的工作不是"等它答对"，是"设计 prompt 让它答对的概率变高"。
  • System prompt 保持 right altitude —— 不过度硬编码（写成 if-else 逻辑）、不过度模糊（"做得好一点"）。在两者之间找平衡。
  • 保持上下文信息密集但紧凑（informative yet tight） —— 不要堆废话（"请仔细思考，认真回答……"），但要信息密度高。每一个 token 都该有价值。
  • 测试 → 观察失败 → 加规则 → 循环 —— 不靠"感觉写得好"，靠"测出来好"。每次加新规则前，先有一个能复现失败的测试用例。

这 5 条原则比 40 个技巧都重要。技巧会过时（Claude 模型在变），原则不会。模型升级后，某些技巧可能不再需要（比如 Extended Thinking 默认开了），但这 5 条原则在 5 年后还是对的。

普通读者怎么用
选一个你反复让 Claude 做的任务。写 5 个测试用例（5 个不同的输入 + 你期望的正确输出）。然后用这 5 个用例跑你现在的 prompt——记录哪些跑错了。错的那些就是你要修的——按本文的 40 个技巧去改。改完再用 5 个用例测。直到全过。这一套流程跑通，你就有了"prompt 工程的肌肉"。

CHEAT SHEET

40 个技巧速查表

按视频章节分组，每条一句话。可以截图保存，或打印贴在桌前。

01 Introduction · 提示词工程定义— What prompt engineering is
  • 01清晰指令 + 必要上下文 + 合理排列——三件套缺一不可。
  • 02练比读有用——学这个最好的方式就是写、改、看输出、再改。
02 Setting the stage · 真实场景设定— Real scenario over toy example
  • 03用真实业务场景——toy example 教不会真东西。
  • 04模型会自信地胡说——"s skiing accident"幻觉是设计问题不是模型问题。
03 Prompt structure · 10 步结构— The 10-step anatomy
  • 0510 步模板——role / tone / background / task / rules / examples / history / request / think / output。
  • 06静态信息放 system prompt——省钱 + 推理稳 + prompt caching 友好。
  • 07动态信息放 user 消息——当前任务、当前文档、当前问题。
  • 08顺序比措辞重要——静态先、动态后；上下文先、查询后。
  • 09System prompt 保持 right altitude——不过度硬编码，也不过度模糊。
04 Task context · 任务上下文— Who Claude is
  • 10明确角色定位——不要只说"helpful assistant"，说具体角色 + 行业 + 专业领域。
  • 11指定语气——"factual and confident" / "warm and conversational"。
  • 12设定回答边界——"不确定时说我不确定"，不要让它自信胡答。
  • 13不指定角色 = Claude 猜你想要什么——猜错就翻车。
05 Background detail · 背景细节— Where to put long context
  • 14区分静态 vs 动态——每次不变的 vs 每次变的，分开放。
  • 15长文档放 prompt 顶部——20k+ tokens 时尤其重要。
  • 16图像可 Base64 编码——直接放进 user 消息的 content 数组。
  • 17多图像场景 Claude 表现好——Opus 4.5/4.6 在多图像理解上有专门优化。
  • 18XML tag 划分背景——<document_1> <document_2> 让 Claude 知道哪是哪。
06 Examples · 少样本学习— Few-shot beats many rules
  • 19少量多样化示例——3-5 个覆盖典型场景，胜过 50 个重复。
  • 20XML tag 包装示例——<example> 让 Claude 知道这是学习样本。
  • 21边缘案例做成示例——不确定 Claude 会不会懂的，做成示例教它。
  • 22不要堆砌 edge case——挑最有代表性的几个，不是把所有可能都列出来。
  • 23示例放在请求前——先看"该怎么做"再看"这次怎么做"。
07 Conversation history · 对话历史— What to carry over
  • 24传相关历史不传全部——切换话题开新对话。
  • 25摘要历史比贴原历史省 token——超 10 轮后摘要早期消息。
  • 26历史相关性过滤——中间工具调用可摘要，核心意图保留原话。
08 Task reminder · 任务提醒 + 分步— Recap + think step by step
  • 27末尾 reminder 抗衰减——长 prompt 末尾把核心规则复述一遍。
  • 28Think step by step——让 Claude 先想再答，准确率提升 10-30%。
  • 29XML tag 区分思考和答案——<thinking> 给过程，<answer> 给结果。
  • 30步骤顺序符合人类推理——先分析 → 再对照规则 → 最后给结论。
09 Output formatting · 输出格式— Structure for parsing
  • 31明确指定输出格式——XML / JSON / Markdown，越具体越好。
  • 32让程序能直接解析——工程化 Claude 的基础。
  • 33XML 嵌套表达层级——比平铺字段清晰。
  • 34格式说明放后段——任务理解在前，格式约束在后。
10 Pre-fill & extended thinking · 预填充 + 扩展思考— Skip small talk, show the work
  • 35预填充跳过废话——API 里预填 Assistant 起始字符串。
  • 36强制输出格式——预填 <verdict> 直接进入正文。
  • 37Extended Thinking 是模型能力——API 里启用，Claude 内部推理更彻底。
  • 38Extended Thinking 适合调试——看推理链路知道哪里想错了。
11 Bonus · 工具与 Agent— From Prompting for Agents
  • 39工具描述清晰独立——不要 6 个相似搜索工具让 Claude 困惑。
  • 40工具错误信息 actionable——告诉 agent 下一步该怎么改，不是返回 stack trace。
