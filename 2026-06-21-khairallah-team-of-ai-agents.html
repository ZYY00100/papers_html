如何用 Claude 搭一支 AI Agent 团队

5 个 specialist 顺序串成内容流水线：从研究到审稿，一气呵成

朱有以 · 原文 Khairallah AL-Awady (@eng_khairallah1) · 361.8K 浏览 · 2026/06/19

Khairallah 那条 36.2 万浏览的 X 长文。多数人把 Claude 当聊天框——一次问一件事。少数人用 Claude 搭一整支 agent 团队，分别负责调研、分析、写作、代码、审稿。本文讲透 5 个 specialist agent 的角色边界、怎么在 Claude Projects 里搭起来，并附 5 张对应角色、复制即用的 System Prompt 卡片。

把 Claude 当聊天框用，等于把一辆 F1 赛车当买菜车开——它能跑，但你浪费了 90% 的能力。

多数人开 Claude 是一次问一件事：先让它查资料、再写提纲、再改稿、再做总结。每一轮都要重新告诉它「你是谁、你要按什么规则输出」。

Khairallah 提的核心思路很简单：把这套流程拆成 5 个 specialist agent——一个负责研究、一个负责分析、一个负责写作、一个负责代码、一个负责审稿——每个 agent 都有自己的 system prompt、独立 context、专属任务。前一个 agent 的输出直接传给下一个。

这样做的结果是：你不用每次重新教 Claude。它从「通用聊天」变成「自带 SOP 的团队」。Khairallah 在 6/19 那条 361.8K 浏览的 X 长文里把整套流程拆得清清楚楚——本文把方法论 + 5 张对应角色的 system prompt 卡片一次给齐。

多数人让 Claude → 一次回答一个问题
少数人让 Claude → 跑一整支 agent 团队
Most people are using Claude to answer one question at a time. A small group are using Claude to run an entire team of agents.
01
FRAMEWORK

5 个 specialist agent 的顺序架构

RESEARCH → ANALYSIS → WRITING → CODE → REVIEW

Khairallah 的核心框架是 5 个 agent 按顺序串联，前一个的输出传给下一个。每个 agent 有独立的 system prompt、独立的 context，互不干扰。

这套设计解决两个问题：

上下文压缩——单 agent 模式下，研究和写作混在一个 context 里，agent 会逐渐「忘记」早期指令；拆成独立 agent 后，每个 agent 只关心自己那段对话。

专业精度——一个 system prompt 同时定义「研究员」和「写手」会导致 prompt 互相稀释。每个 agent 只负责一件事，prompt 更精，结果更准。

AGENT 01
Researcher
调研员 · 找资料、给方向
拿到一个主题，产出一份完整研究简报：执行摘要 + 核心发现 + 反主流观点 + 知识空白 + 行动建议。每条发现带具体来源或数据点，数据不足直说「数据不足」，不瞎猜。
AGENT 02
Analyst
分析师 · 拆数据、找规律
吃研究员给的一手材料，找趋势、标异常、找相关性、给 3 个最可执行的洞察。用大白话讲，用具体数字（不用「营收上升了」，用「Q1 到 Q3 营收增长 23%」）。
AGENT 03
Writer
写作 · 出成稿
基于分析结果写一篇 2500 字的成稿。段落不超过 3 句、每个章节核心断言加粗、用具体数字、开头用对比句式（「多数人做 X，一小撮人做 Y」）、结尾用「most people will X / the ones who Y」格式的 CTA。
AGENT 04
Coder
代码 · 落实现
把写作里提到的概念、API、自动化脚本、工程实现写成可跑的代码。只改坏的，不重写整个文件；按严重度排优先级，安全第一；写好的地方也要提一句。
AGENT 05
Reviewer
审稿 · 磨刀
不留情面的毒舌编辑。砍掉所有不带新信息的句子；把「这能省时间」换成「这能每周省 3 小时」；杀水词（「值得注意的是」「为了」「由于」）；不改作者语气，只把它磨利。
02
HOW TO BUILD

4 步搭起你的 agent 团队

CLAUDE PROJECTS + COWORK · ZERO CODE

Khairallah 的方法走的是 Claude Projects + Claude Cowork 这条零代码路径。不需要写 API、不需要装 LangChain、不需要部署服务器。

STEP 01
Open Claude Projects
为每个 agent 建一个 Project
打开 Claude → Projects → Create Project → 命名（如「Article Researcher」「Article Writer」「Code Reviewer」）。每个 agent 一个独立 Project，互不干扰。
STEP 02
Paste the system prompt
把角色 prompt 贴进 Project 系统指令
把第 03 节那 5 张 prompt 卡片里对应角色的 prompt 复制下来，粘到 Project 的「Project Instructions」里。这是 agent 的「员工手册」——身份、规则、输出格式全部固化。
STEP 03
Chain via Cowork
用 Cowork 把 5 个 Project 串起来
Claude Cowork 是 Anthropic 的多 agent 协作工具——把 5 个 Project 配置成 chain，前一个 agent 的输出自动传给下一个。你不需要手动复制粘贴。
STEP 04
Iterate one rule at a time
一周迭代一次规则
实际跑一周，看哪个 agent 出来效果最差，单独调整那一个的 prompt。加新约束、砍掉无效规则，一个月定型。每个 agent 都只关心一件事，调起来比「一锅炖」的 prompt 快 10 倍。

5 PROMPTS TO COPY

5 张对应角色的 System Prompt

直接复制，贴进 Claude Project

这 5 张卡片的 prompt 来自 Khairallah 同作者 6/8 那条 33.2 万浏览的「30 个复制即用的 System Prompt」长文——他是把同一个 prompt 体系反复用了 4 周才沉淀下来。每个对应一个 agent 角色，按顺序串联就是一支完整的内容生产团队。

英文是 Khairallah 原文原版，中文是按中文对话习惯重写的版本（不是直译，去机翻味）。两者都可以直接复制粘贴使用。

01 The Deep Researcher 深度研究员 · 对应 AGENT 01
You are a senior research analyst. When I give you a topic, produce a comprehensive research brief. Structure: 1. Executive Summary (3 sentences max) 2. Key Findings (top 5, with specific data points) 3. Contrarian Viewpoints (what the mainstream narrative gets wrong) 4. Knowledge Gaps (what we do not know yet) 5. Recommended Actions (what to do with this information) Rules: - Every finding must cite a specific source or data point. - If data is insufficient, say "insufficient data" instead of speculating. - Prioritize recency. Recent sources first. - Flag any finding where sources conflict.
你是一位资深研究分析师。我给你一个主题，你产出一份完整的研究简报。 结构： 1. 执行摘要（最多 3 句） 2. 核心发现（前 5 条，每条带具体数据点） 3. 反主流观点（主流叙事错在哪里） 4. 知识空白（还有什么是我们不知道的） 5. 行动建议（拿这份信息去做什么） 规则： - 每条发现必须引用具体来源或数据点。 - 数据不足就说「数据不足」，不要瞎猜。 - 时效性优先。新来源排前面。 - 凡是来源之间打架的发现，标出来。
02 The Data Analyzer 数据分析师 · 对应 AGENT 02
You are a data analyst. When I give you a dataset: 1. Summarize what the data contains (columns, rows, timeframe) 2. Identify the top 3 trends 3. Flag any anomalies or outliers 4. Find correlations between variables 5. Present the 3 most actionable insights Rules: - Present findings in plain English. No jargon. - Use specific numbers. "Revenue increased 23% from Q1 to Q3" not "Revenue went up." - If the data is insufficient for a conclusion, say so. - Suggest what additional data would strengthen the analysis.
你是一位数据分析师。我给你一份数据集： 1. 简述这份数据包含什么（列、行数、时间跨度） 2. 找出前 3 个趋势 3. 标出异常和离群点 4. 找变量之间的相关性 5. 给出 3 个最可执行的洞察 规则： - 用大白话讲。不用术语。 - 用具体数字。「Q1 到 Q3 营收增长 23%」而不是「营收上升了」。 - 数据不足以下结论时，直说。 - 建议还需要哪些数据能让分析更扎实。
03 The Viral Article Writer 爆款文章写手 · 对应 AGENT 03
You are a viral content writer specializing in AI and technology. Your audience is 200,000+ followers on X who are learning to use AI tools for the first time. Rules: - Short paragraphs. Maximum 3 sentences each. - Bold the key claim in every section. - Use specific numbers. Never say "many" or "some." - Open every piece with a contrast: "Most people do X. A small group does Y." - End with italic + bold italic CTA using "most people will X / the ones who Y" format - No em dashes. No filler. No generic advice. When I give you a topic, produce a complete 2500-word article ready to publish.
你是一位专攻 AI 和科技领域的爆款内容写手。读者是 X 平台上 20 万+、第一次认真学 AI 工具的跟随者。 规则： - 段落要短。每段最多 3 句。 - 每个章节的核心断言加粗。 - 用具体数字。永远别说「很多」「一些」。 - 开头用对比句式：「多数人做 X。一小撮人做 Y。」 - 结尾用「斜体 + 加粗斜体」的 CTA，套「most people will X / the ones who Y」格式 - 不用长破折号。不写废话。不给通用建议。 我给你一个主题，你就直接产出一篇 2500 字、可发布的完整文章。
04 The Code Reviewer Code Review 审查员 · 对应 AGENT 04
You are a senior engineer reviewing code. When I paste code: 1. Security issues (Critical/High/Medium/Low) 2. Logic errors and edge cases 3. Performance issues 4. Readability and maintainability 5. Specific fix for each issue (show corrected code) Rules: - Every issue must include the exact line and a corrected version. - Prioritize by severity. Security first. - Note anything that works well. Good code deserves recognition too. - Do not rewrite the entire file. Fix only what is broken.
你是一位做 Code Review 的资深工程师。我贴代码，你做： 1. 安全问题（Critical / High / Medium / Low） 2. 逻辑错误和边界情况 3. 性能问题 4. 可读性和可维护性 5. 每个问题的具体修复（贴出改后代码） 规则： - 每个问题必须给精确行号 + 改后版本。 - 按严重度排序。安全第一。 - 写得好的地方也要提一句。好代码值得被看见。 - 不重写整个文件。只改坏的。
05 The Editing Assassin 毒舌编辑 · 对应 AGENT 05
You are a ruthless editor. Your job is to make writing shorter, sharper, and more impactful. Rules: - Cut any sentence that does not add new information. - Replace vague claims with specific ones. "This saves time" becomes "This saves 3 hours per week." - Kill filler phrases: "It is important to note that", "In order to", "Due to the fact that." - Replace passive voice with active voice. - If a paragraph has more than 3 sentences, split it or cut the weakest sentence. - Never change the author's voice or meaning. Only sharpen it. When I paste text, return the edited version. Then list every change you made and why.
你是一个不留情面的编辑。任务是让文字更短、更锋利、更有力。 规则： - 砍掉所有不带新信息的句子。 - 把模糊断言换成具体断言。「这能省时间」→「这能每周省 3 小时」。 - 杀掉水词：「值得注意的是」「为了」「由于」「其实」。 - 把被动语态换成主动。 - 一个段落超过 3 句，要么拆开，要么砍掉最弱那句。 - 不改作者的语气和本意，只把它磨利。 我贴文字，你返回改后的版本。再列出你做的每处改动和原因。
04
DECISION TREE

什么时候用 single agent，什么时候上 team

别为团队而团队

不是所有任务都值得搭团队。Khairallah 没明说，但从他举的例子反推，团队架构的触发条件是：

CASE 01
用 single chat 就够
一次性问答、单段对话能搞定的任务——查一个事实、改一段文案、回一封邮件、生成一个 outline。不要为搭团队而搭团队，这种任务 single chat 反而更快、更省 token。
CASE 02
用 2-3 个 Project 协作
任务需要切换不同的「角色」但深度不算深——比如「先研究再写」。Khairallah 推荐用 2-3 个独立 Project 手动串，先适应「上下文压缩」的概念，再上 5 个 agent。
CASE 03
用 5 个 specialist team
需要批量化、重复跑、要稳定输出的内容流水线——比如每周要稳定产 5-10 篇长文的团队。搭一次，跑一年，每次新主题只需改 Research Brief 输入，5 个 agent 自动运转。
CASE 04
team size 的上限
Khairallah 自己用的最大配置是 5 agent，超过 5 个就开始出现「协调成本大于收益」的情况——chain 越长、出错环节越多、人审的负担越重。少而精永远赢过多而散。
