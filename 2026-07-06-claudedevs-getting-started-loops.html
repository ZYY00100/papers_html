Claude Code 团队把「loop」定义为 agent 重复工作直到满足停止条件 —— 按触发方式、停止方式、所用 primitive、适用任务分成 4 类:Turn-based / Goal-based / Time-based / Proactive。本文是官方文章的逐段中文解读,4 段可直接复制的命令 + 3 张原文配图 + 一张总结表都留给你。

原文封面 · Delba Oliveira(@delba_oliveira) · ClaudeDevs · 2026/07/06

最近大家都在谈「设计 loop」而不是「提示词工程」。如果你花点时间在 X 上翻，会发现 loop 这个词每个人说的都不一样。

在 Claude Code 团队，我们把 loop 定义为:agent 重复执行同一段工作，直到满足停止条件为止。我们按 4 个维度给它分类:

  • How they are triggered
  • How they are stopped
  • What Claude Code primitive is used
  • What type of task is most appropriate for each.

我们会讲主流的 loop 类型、什么时候用哪种、怎么在管好 token 的同时不丢代码质量。不是每个任务都需要复杂的 loop —— 从最简单的方案开始，按需用这些 pattern。

On the Claude Code team, we define loops as agents repeating cycles of work·until a stop condition is met.
Delba Oliveira · on what a loop actually is

02
LOOP TYPE 1

Turn-based loops

回合制循环(每条 prompt = 一个手动 loop)

你每发一条 prompt，就已经开了一个手动 loop —— 你自己指挥每一回合。Claude 收集上下文 → 动手 → 检查 → 不行就再试 → 给你回复。我们管这叫 agentic loop。

比如让 Claude 写一个点赞按钮。它会读你的代码、动手改、跑测试，回来一份「我觉得可以了」的版本。然后你人工看一下，再发下一条 prompt。

你可以把这步验证自己做的事写成一个 SKILL.md，让 Claude 能端到端自检。要带工具或连接器，让 Claude 能「看到、测量或触碰」结果。检查越量化，Claude 自检越容易。

比如你的 SKILL.md 里可以这么写:

01 verify-frontend-change SKILL.md 前端变更五步验证
--- name: verify-frontend-change description: Verify any UI change end-to-end before declaring it done. --- # Verifying frontend changes Never report a UI change as complete based on a successful edit alone. Verify it the way a human reviewer would: 1. Start the dev server and open the edited page in the browser. 2. Interact with the change directly. For a new control (button, input, toggle): click it, confirm the expected state change, and screenshot before/after. 3. Check the browser console: zero new errors or warnings. 4. Use the Chrome Devtools MCP, run a performance trace and audit Core Web Vitals. If any step fails, fix the issue and rerun from step 1 — do not hand back partially verified work.
# verify-frontend-change · SKILL.md(中文意译版) ## 作用 把这个 SKILL 放在你的项目里，Claude 改完任何 UI 都要按这 5 步自检，不能只看到「edit succeeded」就宣布完成。 ## YAML 头 --- name: verify-frontend-change description: 端到端验证任何 UI 变更，不能只凭「编辑成功」就宣布完成 --- ## 主流程 永远不要只因为「编辑成功了」就报告一个 UI 变更已完成。要像真人 reviewer 那样去验证: 1. 启动 dev server，在浏览器里打开被改动的页面。 2. 直接跟这个变更交互。如果是新加的控件(button / input / toggle): 点一下，确认状态变化符合预期，截图保存改动前后对比。 3. 看浏览器 console:零个新的 error 或 warning。 4. 用 Chrome Devtools MCP，跑一次 performance trace，审一下 Core Web Vitals。 任何一步失败，修一下，然后从第 1 步重跑 —— 不要带着半验证的工作回来。

FOUR METADATA

触发由用户 prompt 触发。

停止条件Claude 自己判断任务完成，或需要补充上下文。

适合短任务，不属于固定流程或排期的一部分。

管理用量写具体的 prompt，用 skill 强化验证，减少回合数。

03
LOOP TYPE 2

Goal-based loop (/goal)

目标制循环(用 /goal 定义成功标准)

有时候一回合不够，尤其是复杂任务。Agent 在能迭代的时候表现更好。你可以延长 Claude 跑多久:用 /goal 定义「做完了是什么样」。

你把成功标准定下来，Claude 就不用自己判断「够好了」而提前结束。每次 Claude 想停，evaluator 模型会核对你的条件，把它打回去继续，直到目标达成或你定的回合数用完。

所以才说「通过的测试数」「分数超过某条线」这种确定性指标特别有效。

例如:

02 /goal Lighthouse 90 用 /goal 把 Lighthouse 打到 90
/goal get the homepage Lighthouse score to 90 or above, stop after 5 tries.
/goal get the homepage Lighthouse score to 90 or above， stop after 5 tries.

FOUR METADATA

触发实时人工 prompt。

停止条件目标达成，或达到最大回合数。

适合有可验证退出标准的任务。

管理用量定具体的完成标准 + 明确的回合上限，比如「试 5 次就停」。

04
LOOP TYPE 3

Time-based loop (/loop and /schedule)

时间制循环(/loop 与 /schedule)

有些 agent 工作是周期性的:任务不变，只有输入在变。比如每天早上总结 Slack 消息。还有些依赖外部系统 —— 简单的接入方式是定时去查，看变了什么。比如一个 PR 可能收到 review 或 CI 挂掉。

对这些情况，你可以让 Claude 跑 /loop —— 按间隔重复一条 prompt。例如:

03 /loop 5m check my PR /loop 5 分钟巡检 PR
/loop 5m check my PR, address review comments, and fix failing CI
# /loop 5m check my PR(中文意译) ## 作用 每 5 分钟跑一次 Claude:让它检查 PR 状态，处理 review 评论，修 CI 失败。 ## 直接复制可用的英文命令 /loop 5m check my PR， address review comments， and fix failing CI ## 中文拆解 - /loop 5m — 每 5 分钟跑一次 - check my PR — 检查 PR 状态(有没有新 review、有没有 CI 失败) - address review comments — 处理新进来的 review 评论 - and fix failing CI — 顺手把 CI 跑挂的地方修了 `/loop` 跑在你电脑上，关机就停 —— 适合个人开发者的持续巡检。

`/loop` 跑在你电脑上，你关机它就停。想搬到云上，用 /schedule 创建一个 routine。

FOUR METADATA

触发指定时间间隔。

停止条件你手动取消，或者工作完成(PR 合了、队列清空了)。

适合周期性任务，或对接外部环境/系统。

管理用量拉长间隔，或者改成事件驱动而不是时间驱动。

把 routine 搬到云上,用 /schedule:

04 /schedule every hour bug triage /schedule 每小时巡检 bug
/schedule every hour: check the project-feedback channel for bug reports. /goal: don't stop until every report found this run is triaged, actioned, and responded to. When fixing a bug, use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them.
# /schedule every hour bug triage(中文意译) ## 作用 每小时跑一次:扫 project-feedback 频道的新 bug 报告，直到这一轮发现的每个报告都 triage、处理、回复。修 bug 时用 workflow 在三个并行 worktree 探索方案，让 judge 模型对抗性 review。 ## 直接复制可用的英文命令 /schedule every hour: check the project-feedback channel for bug reports. /goal: don't stop until every report found this run is triaged， actioned， and responded to. When fixing a bug， use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them. ## 中文拆解 - /schedule every hour — 每小时跑一次(云端 routine，电脑关了也能跑) - check the project-feedback channel for bug reports — 扫反馈频道的新报告 - /goal: don't stop until ... — 加目标约束:每个报告都要处理完才能停 - workflow to explore three solutions in parallel worktrees — 修 bug 时并行试 3 个方案 - have a judge adversarially review them — 让 judge 模型对抗性 review，挑最好的 把 routine 搬到云上，即便本机关了它也跑。
05
LOOP TYPE 4

Proactive loops

主动循环(组合多个 primitive)

上面那些 primitive，加上 Claude Code 其他特性(auto mode、dynamic workflows，目前 research preview)可以组合成一个 loop，跑长时间任务。

比如处理进来的反馈，可以这样组合:

  1. `/schedule`(research preview)跑一个 routine，检查新进来的报告
  2. `/goal`定义「做完了是什么样」，用 skill 写清楚怎么验证
  3. Dynamic workflows 编排一组 agent:每个报告 triage、修掉、review 修法
  4. Auto mode 让 routine 不用停下来问你能不能继续

拼起来，一条 prompt 长这样:

FOUR METADATA

触发事件或排期，无人在场。

停止条件每个任务达到自己的目标就退出。整个 routine 在你关掉之前一直跑。

适合重复且定义清晰的工作流 —— bug 报告、issue triage、迁移、依赖升级等等。

管理用量把 routine 路由到更小更快的模型，只在需要判断时才上最强的模型。

06
OPERATIONS

Maintaining code quality

维持代码质量

一个 loop 输出质量的好坏，取决于它周围的系统。设计系统的时候:

  1. 代码库本身要干净:Claude 会跟随代码库里已有的 pattern 和规范。
  2. 给 Claude 一个自检的方法:用 skill 把「什么是好的」写清楚给你和你的团队。
  3. 文档要容易拿到:框架和库的文档跟得上最新最佳实践。
  4. 用第二个 agent 做 code review:一个全新上下文的 reviewer 更少偏见，不会被原作者的推理带跑。你可以用内置的 `/code-review` skill，或者用 Code Review for GitHub。

当某次单独的结果不达标，别只修这一处 —— 把它编码成规则，让未来每一轮都受益。

07
OPERATIONS

Managing token usage

管理 token 用量

管 token,loop 要有清晰的边界:

  1. 挑对的 primitive 和模型:小任务不需要多个 agent 或 loop。有些任务可以上更便宜更快的模型。
  2. 定义清晰的成败和停止标准:说清楚「做完了是什么样」,Claude 就能更快收敛到解(但不能太快)。
  3. 大规模跑之前先小规模试:Dynamic workflows 能并行起几百个 agent。先用一小片工作估一下用量。
  4. 确定性工作用脚本:跑脚本比从头推演便宜。比如一个 PDF skill 可以带一个填表单的脚本，让 Claude 每次跑，而不是重新推导代码。
  5. routine 别跑得比实际需要还勤:间隔跟「你监控的东西变化频率」对齐。
  6. 用量复盘:`/usage` 命令按 skill / subagent / MCP 拆解最近用量;`/goal` 不带参数会显示回合数和已用 token;`/workflows` 显示每个 agent 的 token 用量，你随时可以停掉一个 agent。
08
GET STARTED

Getting started

入门指引

总结一下:

想上手 loop，从你已经在做的工作里挑一个。挑一个你是瓶颈的任务，问自己哪一段可以交给 Claude:你能写出验证吗？目标够清晰吗？工作是按排期来的吗？

有想法之后，跑一下 loop，观察结果(看哪里卡住、哪里跑过头)，别怕迭代。

更多信息，看 Claude Code 关于并行跑 agent 的文档，以及 loop、schedule、goal、dynamic workflows 几个页面。
