What is a loop, and how it differs from a prompt

什么是 loop —— 跟 prompt 的本质区别

之前一篇 5/22 的文章（解读 Anatoli Kopadze 那条 101K 浏览的 AI Agents 长文）讲过：agent = chat + tools + memory + loop。在 Anatoli 那篇里 loop 还只是 agent 的一个内部组件。

steipete 这次说"loop 是你设计的"——意思不同：loop 不该藏在 agent 内部，应该是你显式设计。这是同一条概念链的第三次升级：

  • Prompt：你问，agent 答。一次性的，你始终在场
  • Agent：agent 自己能调 tool、能记 memory、能在 loop 里迭代——但 loop 是 agent 内置的
  • Loop engineering：loop 是你写的小程序。你不在 loop 里，你是 loop 的作者

把这件事讲透的，是 Anthropic Claude Code 团队负责人 Boris Cherny：

BORIS CHERNY · ANTHROPIC"I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops."

"我的工作就是写 loop"——这句话的份量在于，说它的人是 Anthropic Claude Code 的负责人。他不是在用 Claude Code，他是在用 loop 来编排 Claude Code。

Addy Osmani's Loop Engineering

Addy Osmani 完整解读 —— 5 个 building blocks + 3 个 loop 还没替你做的事

Addy Osmani（Chrome team lead，工程界老牌意见领袖）6/7 当天发了完整长文《Loop Engineering》，副标题是 "From prompting agents to designing the systems that prompt them."

Addy 把一个 loop 拆成 5 个 building blocks：

  • Prepare context —— 任务 + 指令 + memory + 历史拼成一个 context
  • Call model —— 把 context 发给 LLM，拿回响应
  • Handle response —— 如果是文字，结束；如果是 tool call，执行
  • Iterate —— 把 tool 结果塞回 context，回到第 2 步
  • Return —— 最终响应准备好，结束

每个 coding agent（Claude Code、Codex、Cursor、Pi）背后都是这 5 块。区别不在结构，在你用什么方式把这 5 块拼起来、让它替你跑多久。

但 Addy 最重要的章节是 "What the loop still does not do for you"——3 件事 loop 还没替你做，而且做得越好这些事越尖锐：

  • 验证仍然在你（Verification is still on you）—— 一个无人值守的 loop 也是一个无人值守地犯错的 loop。loop 里的 verifier sub-agent 是为了让 "its done" 有意义，但 "done" 是声明不是证明。你的工作还是"交付你确认过能跑的代码"
  • loop 改变了工作，没把你从工作中删除（The loop changes the work, it does not delete you from it）—— 你的角色从"敲键的人"变成"设计 loop 的人 + 验收的人"，但你不能消失
  • Token 成本（Token cost can be huge）—— loop 越自动，token 烧得越快。sub-agent、verifier、retry 都在烧 token。"token rich" 和 "token poor" 的用法会差别巨大
ADDY OSMANI"Verification is still on you. A loop running unattended is also a loop making mistakes unattended. The whole reason you split the verifier sub-agent from the maker is to make the loop's 'its done' mean something, and even then 'done' is a claim and not a proof."

What one loop looks like

一个 loop 长什么样 —— Addy 的 control panel 例子

Addy 给了一个非常具体的例子：把 5 个 building blocks 粘起来后，一个 thread 变成一个 control panel。他用的形状大致是：

CONCRETE SHAPE
一个 loop 跑起来后，输出不是"一段回答"，而是这些你能在 UI 里看到、能在 CI 里跑、能在 webhook 里推的东西的组合：
— compile / lint / type-check 自动跑，错误回灌到 context
— test 自动跑，失败的 case 重写 prompt 再来
— screenshot / DOM diff（web 项目）让 agent 看见自己改完长什么样
— verifier sub-agent 用一个不同的模型 review 一下这次的输出
— retry 策略：失败 N 次后降级，或者把任务扔回给人类
你设计的不是 prompt，你设计的是这一整套"agent 的环境"

Addy 强调一个反直觉的点：这些步骤你只设计一次。之后你不再 prompt 这些步骤，loop 自己跑。steipete 那条金句的真正含义就在这——你的一次性投入换来的是之后的不在场。

这也是 Boris Cherny 说 "My job is to write loops" 的意思：写 loop 是新工作，prompt 是旧工作。不是 prompt 没用，是它的天花板太低了。

把 steipete 自己的回复提炼出 4 个关键判断：

  • 跳跃期 6 个月 —— 从"写函数"到"设计写函数的 loop"，多数人卡 6 个月。预期管理：这是技能升级，不是新工具
  • 你仍然要在 loop 里（人话版）—— 除非 token 无限，否则你得测试。Addy 说的"验证仍然在你"在 steipete 这边变成"你在 loop 里，但角色是验收"
  • 多 agent 管理是真实需求 —— steipete 自己做的 AgentBox 用来管理 tens of agents，跑 Claude / Codex / Opencode 多个工具混用
  • 下一步是 fleet of loops（@gauthampai 提的）—— 现在你写一个 loop；未来 loop 多到需要 fleet 调度。"fleet" 暗示的是一种类似 Kubernetes 之于容器、Airflow 之于 DAG 那种"调度层"会出现
