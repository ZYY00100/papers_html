硅谷新岗位 FDE 是什么？Andrew Ng 一条推讲透 AI Engineer 才是大机会

by @朱有以 AI / 职业趋势 2026-06-02

OpenAI 和 Anthropic 都在抢一种叫 FDE 的新岗位。Andrew Ng 说他自己的组织也在招，但招的数量远远不如另一种岗位：AI Engineer。

一条推文，凭什么值得花一篇文讲

2026 年 5 月 29 日，Andrew Ng 在他的 The Batch 周报上发了署名信，标题叫 Forward Deployed Engineers and the Future of AI Engineering。同一条内容也发到了 X 上，被算法推到不少人的首页。

Ng 把这个岗位拆得很清楚：硅谷正在疯抢一种叫 FDE 的新工程师，但 AI Engineer 才是真正的大盘。读完之后我对这件事的理解是——FDE 是 AI 商业化的"特战队"，而 AI Engineer 是 AI 落地到每一家公司的"主力军"。两个都重要，但数量级不在一个层面。

这篇文章先把 FDE 这个岗位讲透，再把 Ng 的核心观点提炼出来，最后给你一份工具推荐和行动指南：如果你想往 AI Engineer 这个方向走，现在该学什么、用什么、怎么开始。

FDE 是什么

FDE = Forward Deployed Engineer，前置部署工程师。一个被厂商派到客户公司内部、深度参与业务的工程师角色。

和普通售前工程师（pre-sales）或现场顾问不同，FDE 不是来"卖"的，是来"干"的——直接坐在客户团队里，用客户的工具、跑客户的数据、调客户的系统，把厂商的 AI 产品真的跑通到能用的程度。

"an engineer who is embedded within a client organization to help customize solutions, such as building and tuning agentic workflows that suit the client's particular needs." —— Andrew Ng, The Batch, 2026.05.29

重点是"agentic workflows"这四个字。FDE 不只是把现成 LLM 装上，而是要构建和调优一套完整的智能体工作流，让 AI 真的在客户的业务流程里跑起来。这意味着 FDE 既要懂技术，也要懂客户的业务。

这个岗从哪来：Palantir 20 年前的发明

FDE 不是 AI 时代的新发明。它的祖宗是 20 年前的 Palantir——那家做政府大数据分析的公司。

当年 Palantir 拿下 CIA、IRS 这些政府客户之后，发现一个现实问题：这些客户的核心系统是物理隔离（air-gapped）的，云上的工程师根本连不进去。怎么办？只能派工程师带着笔记本去客户现场，长期驻场开发。这就是 FDE 最早的形态。

这个角色之所以"活"下来，是因为它解决了一个真实痛点：客户买了软件不等于能用上。从软件交付到真正能跑出业务结果之间，隔着一道巨大的鸿沟。FDE 就是专门填这道鸿沟的。

现在 AI 时代这道鸿沟更大了——LLM 拿到手是一回事，把它调成能解决你公司具体问题的 agentic workflow 是另一回事。所以 FDE 这个角色又火了。

现在谁在大规模招：OpenAI 和 Anthropic

Ng 在原文里点了两家公司的名：

⬡

OpenAI · Deployment Company

2026 年 5 月新组建的部门，专门派工程师到企业客户内部做部署和调优。把 OpenAI 的模型和工具真正装到客户的生产环境里。

▲

Anthropic · 企业 AI 服务公司

联合 Blackstone、Hellman & Friedman、Goldman Sachs 成立的合资公司，专门为大型企业提供 AI 落地服务，FDE 是核心角色之一。

两家头部 AI 公司几乎同时把 FDE 提到战略级——这不是巧合，是市场信号：AI 公司终于承认，光卖 API 和模型不够，必须派人到客户现场才能真正把生意做大。

FDE 每天到底在做什么

Ng 列出了 FDE 的核心能力组合，比一般工程师多了一项：商业沟通。

1

听懂客户的真实需求

客户说的"我想要一个 AI 助手"通常不是真的需求。要追问场景、追问流程、追问痛点，把模糊需求翻译成可执行的技术方案。

2

排优先级 · 做策略

客户的需求永远做不完。FDE 要帮客户决定先做什么后做什么，哪些是 80 分够用、哪些必须 95 分。这本质上是 PM 的活。

3

把复杂技术讲清楚

客户的高管不懂 RAG、不懂 evals、不懂 token 成本。FDE 要能用非技术语言解释为什么这个方案好、那个方案不行。

4

必要时 push back

客户会提不切实际的需求，比如"能不能让它 100% 准确"。FDE 要能专业地、尊重地拒绝，而不是照单全收。

沟通能力 商业判断 技术深度

简单总结：FDE = 高级工程师 + 半个 PM + 半个咨询顾问。能干这个活的人，综合素质要求其实比纯架构师还高。

Ng 的核心观点：AI Engineer 才是真正的大盘

FDE 在火，但 Ng 说他自己虽然也招 FDE，招得"远远不如"另一种岗位多——AI Engineer。

"I believe there will be far more AI Engineer jobs than FDEs." —— Andrew Ng

他给了三个理由，每一个都很扎实：

理由一：一家公司能接受的 FDE 数量有限

FDE 是嵌入到客户公司的外来者，再怎么融合，3-5 个也就到顶了。但客户公司自己会需要几十个、上百个 AI Engineer 来做内部项目。这个数量级差异是结构性的，不是趋势性的。

理由二：vendor 中立性是个真问题

FDE 本质上是厂商派来的——他是 OpenAI 的人，去你公司部署 OpenAI 的方案；他是 Anthropic 的人，去你公司部署 Anthropic 的方案。这就带来一个隐患：你公司的流程会被深度绑死在某一家厂商的技术栈上。

现在这个时点，Ng 用了一个词叫 "optionality"（可选性）——明年谁家模型最强，今年没人知道。把自己的流程绑死在 OpenAI，半年后 Anthropic 出了更合适的模型，想换？整个 agentic workflow 推倒重来。

所以很多客户开始警惕：FDE 可以用，但要警惕你把整个公司的 AI 未来赌在一家厂商身上。

理由三：AI Engineer 角色会再分化

Ng 提了一个很值得记住的类比：现在的 AI Engineer 就像 2000 年前后的"Software Engineer"——一个泛称，什么都干。等这个角色成熟了，它会像当年的 Software Engineer 一样，分化成前端、后端、移动端、数据工程、DevOps 等等。

他列了一些可能的新岗位名字：

  • AI FDE — 专做 agentic workflow 落地的工程师
  • LLMOps Engineer — 负责 LLM 服务的部署、监控、成本控制
  • Evals Engineer — 专门做模型/Agent 评估体系的人
  • AI Data Engineer — 处理 RAG、微调所需的数据管道
  • Harness Engineer — 构建和调优 agent 的"驾驶舱"（tool use、prompt routing、guardrails）

这些岗位现在名字都没统一，但方向已经清晰了——AI 工程师不是一个终点，是一个起点。

optionality 这个隐藏陷阱

Ng 这条推文里最值得多看两遍的概念是 optionality，可选性。

他的原话翻译成大白话："在现在这个很难预测一年后哪家的 AI 服务最强的时点，让自己保留'未来选最适合的那家'的能力，是非常值钱的。"

反过来，让 FDE 把你公司的流程深度绑死——这件事本质上是在用未来的灵活性换眼前的便利。这笔账怎么算都不划算。

这个观点对我们普通人的启发是：选 AI 工具时，不要让自己被锁死。能用 MCP 协议（Model Context Protocol）就用 MCP，能用标准化 API 就用标准化 API，不要一上来就把所有业务逻辑写进某个厂商独有的 SDK 里。

谁适合去 FDE 岗

FDE 不是新人的第一站。如果你正在考虑要不要投这个岗，Ng 的描述其实给出了画像：

维度 FDE 画像 典型来源
技术深度 能独立搭 agentic workflow，懂 LLM、evals、tool use 3-5 年后端 / ML 工程师
沟通能力 能跟 CXO 开会、能给非技术高管讲清楚方案 做过 on-call demo、写过技术博客
商业感觉 能判断哪些需求该做、哪些是 nice-to-have 做过 PM 活、独立 owner 过项目
心理承受 愿意长期驻场、能在客户和厂商之间做翻译 咨询公司出身、独立咨询师

反过来，如果你更愿意自己埋头写代码、不喜欢跟人解释，FDE 可能会让你痛苦。Ng 自己组织招的 AI Engineer 数量比 FDE 多得多，正是因为大部分工程师的画像更适合后者。

工具推荐：AI Engineer 路径该掌握什么

从 Ng 原文里提炼，加上 2026 年现状，给一份入门工具清单。优先选 vendor 中立、迁移成本低的那一类。

⌘

Coding Agents

Ng 在原文里点了四个名字：Claude Code、Codex、Antigravity CLI、OpenCode。这四家是当前 coding agent 的第一梯队，能力接近、风格不同。挑一个用熟，再去试另一个。

Claude Code Codex Antigravity CLI OpenCode
◉

LLM Prompting

prompt 工程师这顶帽子早就被嘲讽了，但 prompt 能力本身是硬功夫。系统提示怎么写、few-shot 怎么给、cot 怎么触发，这些是所有 agentic workflow 的地基。

Claude API OpenAI API Anthropic Console
◇

Agentic Frameworks

能帮你快速搭起 multi-agent 系统、tool use、memory 管理的框架。挑一个学透，不要每个都摸一遍。

LangGraph CrewAI Autogen MCP Servers
✓

Evals

Ng 在原文里特别提了这个词。Evals 是评估 agent 表现的系统化方法——没有 evals 的 agent 跟没有单元测试的代码一样脆弱。

Braintrust Langfuse Promptfoo

原则：优先选 vendor 中立的工具。MCP 协议、标准化 API、开源框架——这些东西能让你随时换底层模型而不用推倒重来。这正是 Ng 强调的 optionality。

行动指南：三步走，从今天开始

不想被时代推着走，就主动跳进去。下面这条路径是给有一定工程基础、想往 AI Engineer 方向走的人。

1

第 1 步 · 第一周：把一个 coding agent 用到熟

在 Claude Code、Codex、Antigravity CLI 里选一个，每天用它干活。不是"试试看"，是"必须用它"。一周后你会有真实体感。

Claude Code Codex Antigravity CLI
2

第 2 步 · 第 2-3 周：搭一个真实的 agentic workflow

找一个你工作里的小任务，用 LangGraph 或 CrewAI 搭一个能跑通的 agent。比如：自动从 RSS 抓内容、自动分类、自动发到 Notion。跑通就是胜利。

LangGraph Tavily Notion API
3

第 3 步 · 第 4 周起：补 evals 和数据基础

同一个 agent，加上 evals 框架。每次改 prompt、改模型，跑一遍 evals 看分数变化。这个习惯会让你在真实工作里少踩 80% 的坑。

Braintrust Promptfoo Langfuse

一个月后，你会有一个能展示的项目、一套习惯的 evals 流程、对一个 coding agent 的深度体感——这是面试 AI Engineer 岗最硬的三件套。

最后说两句

FDE 这个岗会继续火，但它本质是 AI 商业化初期的"特战队形态"——靠精英小队去啃硬骨头。当 AI 真正渗透到每一家公司之后，扛把子的是大量分布式的 AI Engineer。

Ng 的判断是："Skilled AI Engineers are in very high demand."（熟练的 AI Engineer 需求极大。）这句话的潜台词是：现在这个赛道不卷，谁先进去谁占位。

你不需要等岗位定义清晰了才动。先把一个 agentic workflow 跑起来，再谈分化。
